Discussion Post: UML and Secure Software Development

UML and Secure Software Development

The Open Web Application Security Project, or OSWAP, is a non-profit organisation founded in 2001 (OWASP, N.D.) Its main goal is to help highlight vulnerabilities with website, in an effort to prevent breaches in security.

In the OSWAP Top 10 list published in 2021, Broken Access Control is the number one vulnerability (OWASP, N.D.) This is often down to developer miss configuration of the access controls. A specific instance of this is Insure Direct Object Reference, or IDOR. It allows users, with little or no expertise, to access information for which they do not have privileges for. If an e-commerce site uses order IDs directly in URLs without implementing proper authorization checks, an attacker could exploit this vulnerability to view or modify other users’ orders. By simply manipulating the order ID in the URL, the attacker might gain access to confidential information or alter the order details of other customers, leading to unauthorized access and potential data breaches.

“It is a common practice for practitioners to draw the system architecture as a structural component or class diagram in order to represent the relationships between the subsystems.” (Pillai, A. B., 2017)

Unified Modelling Language, or UML, is modelling language that allows the visualisation of a system. It can show software or hardware systems and is used throughout the SDLC. It was adopted by the Object Management Group, or OMG, in 2007 and is now maintained and developed by the OMG. UML has two main sections, Structural diagrams and Behavioural diagrams. (UoEO, N.D.)

Structural diagrams:

  • Class Diagram,
  • Object Diagram,
  • Component Diagram,
  • Composite Structure Diagram,
  • Deployment Diagram,
  • Package Diagram.

Behavioural Diagrams:

  • Use Case Diagram,
  • UseActivity Diagram,
  • UseSequence Diagram,
  • UseCommunication Diagram,
  • UseState Diagram,
  • UseInteraction Overview Diagram,
  • UseTiming Diagram.

In developing an application, several UML diagrams are particularly valuable. The Use Case Diagram is essential for capturing high level functional requirements and interactions between users and the system. The Class Diagram provides a clear definition of the static structure, detailing the classes, attributes, methods, and their relationships within the data model. The Sequence Diagram is crucial for illustrating the detailed interactions and the sequence of messages exchanged operations, ensuring that the process flow is well-understood. The Activity Diagram helps visualize the workflows and processes involved in various operations, offering insights into the sequence of activities and decision points necessary for efficient execution.

Insecure design emphasises on risks around mistakes made during the software design and the software architecture that can be expressed as “missing or ineffective control design” (OWASP Foundation, 2021). More broadly speaking, insecure design pertains to the practice of developing software solutions where the design of the system does not adhere to secure design patterns and best-practice architectural methodologies. This results to the creation of weaknesses which a bad actor could abuse.

In the second figure, the design of the system includes a safeguard against abusing the booking system by enforcing that a deposit is requested regardless of the number of tickets in the reservations. Flowcharts and activity diagrams can be used as means of producing threat modelling analysis for modelling the system, identifying weak points and developing countermeasures.

Some say that people are the biggest risk of cyber security.

“Human risk is today’s biggest Cybersecurity gap and remains largely unaddressed.” (Mimecast, 2024). Human error is the largest growing cyber security threat today, most of these could be mitigated through education. From ISO/IEC Standard 27000 Section 3 the following are five key concepts to managing the risk.

  • 1. Access Control (3.1):
  • Access control makes sure that only authorised personnel should have access to specific systems and information. Having and enforcing strict access control can help to mitigate this security risk. A regular review and updates of the policy helps to enhance security further.
  • 2. Information Security Policy (3.5):
  • Information Security Policy should clearly outline acceptable use, security practices, and the consequences of violations. Training sessions should be conducted to ensure understand and adhere to these policies.
  • 3. Security Awareness, Education, and Training (3.13):
  • These programs should include training on recognising phishing attempts, proper data handling procedures, and the importance of reporting suspicious activities. Educated employees are more likely to follow best practices.
  • 4. Incident Management (3.8):
  • An Incident Management process ensures that insider threats are found and addressed. Users should know how to report potential security incidents, and there should be a clear protocol for investigating and responding to these reports.
  • 5. Monitoring (3.10):
  • Monitoring of systems and user activities allows detection of suspicious behaviour. These tools can log and analyse user actions, flagging anomalies that may show insider threats. Regular audits of these logs can help in finding patterns that suggest malicious intent or negligent behaviour.

As cyber threats evolve, and attacks become more sophisticated, human error stays the largest threat. This points to a need for continued and enhance education to manage insider risk.

What factors determine whether a programming language is secure or not?

  • Memory Safety: Languages that prevent buffer overflows, null pointer dereferences, and memory leaks are considered more secure. This feature is crucial in preventing vulnerabilities like buffer overflow attacks, which are common in languages like C and C++.
  • Type Safety: Strongly typed languages can prevent unintended data manipulation by enforcing strict rules on how data types are used. Type mismatches can lead to vulnerabilities, and strongly typed languages ensure that variables are used correctly.
  • Error Handling: Secure languages provide robust mechanisms to handle errors and exceptions gracefully. Proper error handling can prevent applications from crashing or entering a vulnerable state due to unexpected inputs or conditions.
  • Sandboxing and Permission Control: Languages like Java that provide built-in sandboxing ensure that code executes in a restricted environment, preventing unauthorized access to system resources.
  • Compiler/Interpreter Security: A secure language should have a well-tested compiler or interpreter that doesn’t introduce vulnerabilities during code execution.
  • Standard Library Security: A programming language's standard library should be secure, with well-vetted functions that avoid unsafe practices like the use of insecure cryptography algorithms or unchecked input.
  • Concurrency Handling: As many modern applications rely on parallel processing, languages that handle concurrency securely can prevent race conditions and data corruption.
  • Community and Support: A large, active community and regular updates help ensure that vulnerabilities are quickly discovered and patched.

Could Python be classed as a secure language? Justify your answer.

Python can be considered relatively secure but with some limitations. It manages memory automatically through its garbage collection mechanism, eliminating the need for direct pointer manipulation, thus reducing the risk of buffer overflows and memory-related vulnerabilities compared to languages like C.

Python is dynamically typed, meaning it doesn’t enforce type checking at compile-time, which can lead to issues when types are mixed unintentionally, though its runtime checks provide a layer of safety by throwing exceptions when types are misused.Python’s robust error-handling system, using try-except blocks, allows for efficient management of errors without crashing programs.

Python's interpreter is regularly updated to patch known vulnerabilities, but being an interpreted language, it is susceptible to injection attacks, although proper coding practices can mitigate these risks. The standard library is extensive and generally secure, though certain modules can introduce security risks if mishandled. Secure cryptographic libraries such as hashlib and ssl provide secure communication.Python benefits from a large community and frequent updates, ensuring that known vulnerabilities are addressed promptly.

Python would be a better language to create operating systems than C. Discuss.

While Python is a highly flexible, general-purpose language, it is not better than C for creating operating systems for several reasons. Operating systems require low-level hardware interaction such as memory management, device drivers, and system calls, which C handles efficiently by providing direct access to memory and system resources. Python abstracts these details, which simplifies development but limits its ability to perform efficiently at the low level required for operating systems.

As an interpreted language, Python is slower than compiled languages like C, and since operating systems need to be highly optimized for performance, C remains superior in terms of speed and resource management. Python’s higher-level abstractions could accelerate development for components that do not require tight resource control.

In conclusion, while Python offers ease of development, C is far more suitable for operating system development due to its performance, low-level access, and mature ecosystem.

Programming Language Concepts - RegEx

1. What is ReDOS and what part do ‘Evil Regex’ play?

ReDOS or Regular Expression Denial of Service is an attack that exploits the inefficiencies of regular expressions. During a ReDOS attack a specifically crafted input is used to exploit the vulnerable regular expressions. These poorly optimised expressions are known as ‘Evil Regex’.

For example, the regex pattern (a+)+ is used to match sequences of one or more 'a' characters, grouped in one or more repetitions. While it may seem useful for matching strings like "a", "aa", "aaa", or "aaaa", it is problematic because it allows excessive backtracking.

When the input string is complex, such as "aaaaaa", the regex engine may attempt various combinations of matching the inner and outer groups, leading to performance degradation and making it vulnerable to ReDOS attacks.

Example of Input Causing ReDOS: Input: "aaaaaaaaaaaaaaaaaaaaa!"

The regex engine would inefficiently backtrack to find matches, severely affecting performance.

2. What are the common problems associated with the use of regex? How can these be mitigated?

Problem Explanation Mitigation
Catastrophic Backtracking Poorly constructed regex can lead to excessive backtracking, significantly slowing execution. Backtracking Control: Use non-greedy quantifiers and atomic grouping to prevent excessive backtracking.
Complexity Regex patterns can quickly become difficult to read, debug, and maintain, especially with deep nesting. Pattern Simplification: Break complex patterns into smaller, modular components for better readability.
Overlapping Patterns Ambiguous or overlapping patterns can increase the number of possible matches, slowing down execution. Disambiguation: Remove overlapping alternatives and use more specific patterns to reduce ambiguity.
Unintended Matches Poorly defined patterns may capture strings that should not be matched, leading to incorrect behavior. Input Validation: Define strict patterns and test with edge cases to ensure accurate matches.

3. How and why could regex be used as part of a security solution?

Regular expressions can be used to enhance security by using them in,

  1. Input Validation: Ensure user input conform to expected formats.
  2. Log Analysis: Parse logs to identify patterns
  3. Data Scrubbing: Remove sensitive information from data sets such as credit card numbers.

Cyclomatic Complexity

Importance of Cyclomatic Complexity in Software Development

Cyclomatic Complexity (CC) was introduced by Thomas McCave in 1976, it is a metric designed to measure the complexity of a programs control flow. McCabe’s goal was to provide a quantitative way to assess how complicated a program logic was by counting the number of independent paths through the code.

    Key Benefits of using CC
  • Maintainability:
    • Software with a high CC becomes increasingly more difficult to maintain or update. Keeping the CC low promotes simplicity and allows for easier to maintain and update the code base and reduces the likelihood of bugs.
  • Testing Efficiency:
    • In testing, the higher the CC the more test cases are needed to fully test a piece of software. By reducing the CC, the testing is simplified ensuring all paths can be tested.
  • Security:
    • The higher the CC the easier it is to include complexities that can lead, and make harder find, security vulnerabilities.

In secure software development, it is crucial to keep Cyclomatic Complexity (CC) low. This helps maintain code simplicity, ensuring that all execution paths are thoroughly tested and not overlooked. Additionally, low CC makes the code easier to maintain and update, reducing the risk of introducing new errors or vulnerabilities.

What is and Ontology

When examining ontology, similarities with OOP become apparent. An ontology serves as a high-level abstraction of an OOP design. It refers to a formal, structured representation of an application, defining the entities and their attributes while linking them to illustrate the structure and relationships between the entities.

Key Components:

  • Classes: General categories or types of entities.
  • Instances: Specific objects or instances of a class.
  • Properties: The attributes and relationships between and within classes.

Unlike OOP, ontologies emphasize structure and relationships, whereas OOP also incorporates behaviour and state management.

Faceted Data

Faceted Data

Faceted Data offers a structured approach to storing, managing and protecting sensitive data. It divides the data into multiple groups or facets. This approach also provides access controls for the data going toward an organisation complying with GDPR.

Pros

  • Access Control – each component only has access to the relevant information.
  • Improved Security – successful attacks can only access data for the one component
  • Better Compliance – helps to meet requirements of GDPR
  • Enhance privacy – Components can be adjusted for different roles in the application

Cons

  • Increase Complexity – implementing faceted data can increase the complexity of the solution
  • Overheads – with increase in complexity come an increase in overheads which can introduce bottlenecks in the system
  • Data Fragmentation – with data stored in multiple places it can lead to redundancy and inconsistency.

Key Features of the System Design

  • Data Facets: Each facet holds its sensitive and public data portions.
  • Role-Based Access Control: The access control manager checks the user’s role and grants access to the appropriate data facet.
  • Separation of Data: Sensitive data and public data are clearly separated, with more sensitive data hidden from lower roles like guests.

Debate: Microservices and Microkernels

A Balanced Future with Monolithic and Modular Architectures.

There has been a lot of discussion around the statement “Torvalds has been proven wrong and it only took nearly thirty years. Microservices and microkernels are the future.” While microservices and microkernels are widely used, I believe that the future is a balance between these newer approaches and the proven monolithic architecture.

Monolithic systems such as the Windows NT architecture evolved from a primarily monolithic design into a hybrid kernel. This design has proven effective in delivering both performance and stability. Microsoft adopted this hybrid approach to combine the efficiency of monolithic systems with some modular features, like device drivers. This shows that even modern large-scale systems continue to depend on monolithic kernels for core functions. Windows has been structured as a mostly monolithic kernel with certain modular elements, demonstrating a balanced approach between the two architectures (Russinovich & Solomon, 2009).

Microkernels and microservices are particularly useful in distributed systems and embed environments, as these systems need to be more flexible and allow for scalability. This does, however, introduce overheads as the code becomes more complex, specifically the communication between the modules.

In conclusion I disagree with the original statements. The success of both monolithic and modular approaches shows that both approaches will continue to exist.
