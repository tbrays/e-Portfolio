Factors which Influence Reusability

"Designing object-oriented software is hard, and designing reusable object-oriented software is even harder" (Gamma et al, 1994). Reusability is a powerful strategy for saving time, reducing costs and improving reliability. By reusing established components, developers can avoid reinventing the wheel and focus on innovation and improving functionality.

Padhy et al. (2018) suggested factors that affect the reusability of code. Below is a prioritised list of these factors.

Name

Reason

Requirements Analysis (RA)

The requirements analysis is important for any project. Properly completed it can ensure the application meets specification and allow for easy adaption of elements between projects.

Architecture-Driven Approach (ADP)

A well defined and reusable architecture allows a project to scale well and accommodate changes.

Design Patterns (DP)

These allow the developer to use tried and tested solutions in new projects. This reduces development time and improves reusability.

Modules in the Program (MP)

When designed well, modules can be used throughout the project, making them very useful for reusability.

Algorithms used in the program (AP)

Many algorithms exist to solve different problems. They can be reused between projects, such as collision detection and path finding.

Test Cases / Test Design (TCTD)

Reusing well written test can save a lot of time, improving the reliability of a project.

Documentation in the Project (DIP)

Documentation is important as it gives insight into the program logic and helps with maintenance and updates.

Knowledge Requirements (KR)

Expertise, reasoning and experience are invaluable but are not a standalone reusability but are applied to other factors.

Service Contracts (SC)

Long term relationships between developer and customer is essential but has less effect on reusability.

Used in the Data Project (UD)

Data is often tailored to specific projects so is less useful for reusability.

Models in the Project (MIP)

Project specific models often need significant adjustment to be used in new projects.

Reusability starts in the planning phase, Requirements Analysis and Architecture Driven Approach form the backbone of reusable software. Design Patterns, Modules and Algorithms provide the ability to reuse code from other projects. Testing, Documentation and service contract support a reliable and adaptable project. By focusing on foundational elements and using reusable elements, software development becomes more efficient and sustainable.

Software reusability is an important concept of object-oriented development"and"Padhy et al. (2018) present eleven key reusability factors in Table 1 of their paper.

Having read the paper, I would offer the following prioritisation, listed from highest to lowest importance:

  1. Requirement Analysis (RA)"- Requirement analysis should have the highest priority as it forms the foundation of any software development project. Without a clear understanding of requirements, it is impossible to create reusable components that will serve their intended purpose effectively."As"Padhy et al. (2018) note,"requirements must satisfy contracts, standards, and formal specifications.
  2. Documentation in Project (DIP)"- Proper documentation is necessary for reusability because it helps other developers to understand and use components. This includes technical documentation and requirement specification analysis created during the SDLC.
  3. Design Patterns (DP)"- Design patterns present solutions to common problems and enhance reusability. When"existing designs can be reused"again for similar requirements, development time is reduced and reliability is improved as the patterns have been previously tested and validated.
  4. Modules in Programme (MIP)"- Modular programming is essential for reusability as it allows components to be easily separated and recombined. Each module should be a self-contained, executable part of the application that serves a specific purpose.
  5. Knowledge Requirement (KR)"- Understanding the knowledge requirements, including experience, ideas, and reasoning behind components, is vital for effective reuse. This factor is ranked lower than documentation because good documentation can help transfer this knowledge.
  6. Architecture Driven Approach (ADP)"- Whilst important for"overall"structure, the architecture approach is less critical for individual component reuse than the above factors. However, it remains significant for ensuring components can work together effectively.

The remaining factors (Service Contracts, Test Cases/Design, Used Data, Models, and Algorithms)"whilst"important, have less direct impact on reusability"in my view"than those listed above. For example, algorithms can be modified more easily than entire architectures, and test cases, whilst valuable, can"be recreated"if necessary.

This prioritisation emphasises the human and organisational factors (requirements, documentation, knowledge) over purely technical aspects, as these tend to be more challenging to recreate or modify later. As Seth et al. (2011) observe, the success of object-oriented programming largely depends on its ability to model real-world problems effectively, which begins with proper requirement analysis and documentation.

In my initial post, I discussed the factors influencing reusability in object oriented programming (OOP). I prioritised these factors based on the idea that reusability begins early in the planning phase, with Requirements Analysis (RA) and an Architecture Driven Approach (ADP) providing the foundation for scalable and adaptable software. I also focused on the importance of design patterns, well structured modules, and reusable algorithms in improving development efficiency and reducing redundancy.

Todd's reply challenged my prioritisation by emphasising Knowledge Requirements. Craig's reply also reinforced the value of experience. While he agreed with my prioritisation of RA he highlighted the importance of experience in influencing design decisions, noting that as developers gain more knowledge, they can make better decisions about what will lead to reusable software (Bourne, 2024).

Reflecting on the feedback, I realised I may have ranked Knowledge Requirements too low. I now see that experience and knowledge should be more prominent in the hierarchy, as they directly impact the reusability of software. Reuse-by-Memory and Reuse-by-Recall highlights how developers with more experience are better able to recall and identify relevant components from previous projects, making them more efficient at reusing code, as experienced developers can more easily recognise reusable solutions compared to less experienced ones who rely on retrieval tools (McCarey et al., 2005)

Overall, the discussion strengthened my understanding of reusability, with differing perspectives on prioritisation. The feedback from Todd and Craig underscored the importance of balancing planning and experience.

UML models

UML and the SDLC

Analysis

Use Case Diagram: Used at this stage to show functional requirements. They show the system from an end users perspective and how users interact with the system.

Activity Diagram: Used to show the flow of activities or processes, again helping to understand how the end users and the system interact.

Design

Class Diagram: Used at this stage to define the system structure by mapping out the classes, attributes and methods, along with class relationships.

Sequence Diagram: Used to model how objects interact with each other in a specific scenario, detailing the sequence of messages passed between them.

State Machine Diagram: Used to represent the states of objects and how they transition from one state to another in response to events

Component Diagram: Used to model the physical structure of the system by showing how components (modules, libraries) are organised and interact.

Implementation

Class Diagram: Used as a reference during development to ensure the code adheres to the designed structure of the system.

Sequence Diagram: Guides the implementation of specific functionalities by describing the order and flow of interactions between objects.

State Machine Diagram: Used to implement state-dependent behaviour, ensuring transitions and states align with the design.

Testing

Use Case Diagram: Helps verify whether all functional requirements defined during the analysis phase have been implemented and covered in test cases

Sequence Diagram: Used to validate the correctness of interactions and message flows between objects.

State Machine Diagram: Ensures the system transitions between states as expected during various scenarios.

Deployment and Maintenance

Deployment diagrams: Used to model the physical deployment of software on hardware components, assisting in installation and setup.

Component Diagram: Helps maintain and update the system by showing how individual components interact and fit into the overall structure.

Think Python: Chapter 19.10

19.10 Glossary

Conditional Expression:

Describes an expression that evaluates to one of two possible outcomes based on a condition.
Useful for concise decision making logic within expressions.

List Comprehension:

A concise way to create lists by combining a for loop and an optional conditional statement inside square brackets.
Enables more readable and compact list creation.

Generator Expression:

Like a list comprehension but uses parentheses instead of square brackets, producing a generator object.
More memory efficient as it generates items one by one rather than all at once.

Multiset:

Represents a set like structure where elements can appear multiple times, and their counts are significant.
Often implemented using collections like collections.Counter in Python.

Factory:

A function designed to create and return objects, often used for encapsulating object creation logic.
Facilitates consistent object creation and can simplify code by abstracting complex initialisation.

You are required to identify three operations which you wish to support in your software, from the perspective of a robot fulfilling a job, and providing capability for them through the use of an object-oriented Python program. The system should support the ability of a human to interact with the system through a front-end command-line interface (CLI).
